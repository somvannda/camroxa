# Video Pipeline Flow & Requirements (Boss Spec)
This document captures the end-to-end pipeline across Music → Image → Video → Merger, plus the current Phase 3 (Video render) requirements so we can keep a consistent full-picture later.

## 1) End-to-End Flow (Future-Proof)
### Flow 1 — Music Generation (Batch-based)
- User selects:
  - Date range (inclusive)
  - Channels (OK list + ALT list)
  - Pairing is by selection order (OK[i] with ALT[i])
  - Songs per batch (Count)
- System generates multiple batches sequentially:
  - Batch = (date × channel-pair)
  - Each batch generates N songs
  - Songs are stored the same as today under each profile folder and date-based run folder.

### Flow 2 — Image Background Generation (Planned)
- For each batch from Flow 1:
  - Generate exactly 1 background image (the “batch background”)
  - Save the background under that batch’s run folder so Flow 3 can find it deterministically.

### Flow 3 — Video Rendering (This Spec: Current + Next)
- For each batch:
  - Render one MP4 per MP3 (so N songs ⇒ N MP4 files)
  - The renderer uses:
    - A background image (see “Background Sources”)
    - A logo image for the channel (from profile settings)
    - Audio-reactive visuals (spectrum + particles) driven by the MP3
  - Output MP4s are written per MP3 folder (see “Output Conventions”).

### Flow 4 — Merger (Already exists)
- Merge MP4 outputs into a final combined MP4 per directory (current feature).
- Later: add “merge per batch” convenience that merges the batch’s rendered MP4 set in order.

## 2) Video Tab (Flow 3) — Functional Requirements
### 2.1 Inputs
- Support selecting inputs in either way:
  - Select MP3 files (multi-select)
  - Select folder(s) and scan for `.mp3` files
- The UI shows:
  - The resolved MP3 list (deduped)
  - Which channel/profile each MP3 belongs to (derived from folder selection rule or manual assignment if needed later)

### 2.2 Background Sources (Current)
In Flow 3 (current stage), backgrounds can be:
- Manual: user selects one background image to use for the render run
- Auto: system picks a background image from the selected MP3 folder

Auto background rules:
- Search in the MP3’s parent folder for a background image file.
- Supported extensions: `.png`, `.jpg`, `.jpeg`.
- If multiple images exist, pick deterministically (sorted by filename, first match).
- If no image exists:
  - Show a clear per-item error OR require manual background selection.

### 2.3 Background Sources (Future, after Flow 2 is implemented)
- Default should become: “Use batch background image” (generated by Flow 2).
- Manual background selection remains as a fallback.

### 2.4 Logo Source (Current + Future)
- Logo image is configured per profile (channel) in Profile settings.
- When rendering a video for a song, the renderer uses the logo for the song’s channel.
- Supported logo formats: `.png` (preferred for alpha), `.jpg`.

### 2.5 Rendering Outputs
- Output is written per MP3 folder:
  - Output folder: `<mp3_parent>\video\`
  - Output file: `<mp3_basename>.mp4`
- Rendering is deterministic for the same input set + template.

### 2.6 Effects Requirements (Phase 1–2 subset)
- Must include:
  - Circular audio spectrum visualizer
  - Background image displayed behind spectrum
  - Particles (audio reactive)
- “Background reacts to music” requirement:
  - At minimum: subtle zoom/shake/pulse tied to bass/beat energy.
  - Future: color grading or glow intensity tied to FFT energy.

### 2.7 Batch Mode & Progress
- The renderer is headless and run as a background task.
- The UI must show:
  - Overall progress (% / completed count)
  - Per-item status (queued/running/done/failed)
  - Logs (stdout/stderr)
- Cancellation:
  - Cancel current item and stop remaining queue.

## 3) Output Conventions & Folder Layout (Recommended)
This keeps everything discoverable for later Flow 2 and Flow 4 integration.

### 3.1 Per MP3 output folder
- `<mp3_parent>\video\`

### 3.2 Per batch output folder (future convenience)
If we later render “per batch” directly, we can also write:
- `<profile_run_folder>\video\`
and symlink/copy as needed.

## 4) Non-Goals (for now)
- No realtime preview/editor required.
- No cloud rendering in the local app.
- No node graph/timeline editing.

## 5) Open Decisions (to resolve when implementing)
- Background auto-pick strictness:
  - Should missing background auto-pick fail the whole run or only that item?
- MP3 → profile/channel association when selecting arbitrary MP3 files:
  - If a user selects MP3s from mixed folders, should we require manual assignment to a channel/profile?
