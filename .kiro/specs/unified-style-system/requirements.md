# Requirements Document

## Introduction

The MusicGenerator desktop application (PyQt6) currently has seven competing style sources that conflict through Qt's cascading specificity rules (inline > global, object-name > class). This specification consolidates all styling into a single master QSS system driven by the existing `QSSGenerator` + design tokens, then rebuilds the application shell (sidebar navigation, custom title bar, and login screen) to match a premium NEXORA AI-style dark dashboard design language.

## Glossary

- **Master_QSS**: The single application-wide Qt Style Sheet string generated by `QSSGenerator` from `ThemeTokens` and applied via `QApplication.setStyleSheet()`
- **Design_Tokens**: The frozen dataclass hierarchy (`ColorTokens`, `TypographyTokens`, `SpacingTokens`, `ShapeTokens`) defined in `tokens.py` that serve as the single source of truth for all visual values
- **Property_Selector**: A QSS selector targeting widgets by Qt dynamic property values (e.g., `QPushButton[uiRole="primary"]`), allowing role-based styling without inline stylesheets
- **Style_Helper**: The module `views/helpers/style_helper.py` providing functions (`set_panel_role`, `set_button_role`, `set_label_role`, `set_field_role`) that assign dynamic properties to widgets
- **Legacy_Theme**: The `python_app/app/theme.py` module containing `build_ui_tokens()` and `build_app_stylesheet()` that duplicates the QSS generator's functionality
- **Inline_Style**: Any direct call to `widget.setStyleSheet(...)` within view files that overrides the global QSS
- **App_Shell**: The structural UI frame consisting of the custom title bar, left sidebar navigation, and content area container
- **Title_Bar**: A custom frameless window header widget containing the application logo, name, notification indicator, and window control buttons (minimize, maximize, close)
- **Sidebar**: A fixed-width left navigation panel (200px) with icon+text nav items, a gradient active indicator, and a user profile section at the bottom
- **Login_Screen**: The authentication view displayed at startup, using a split-panel layout with a branded left section and a form right section

## Requirements

### Requirement 1: Remove Legacy Theme Module

**User Story:** As a developer, I want a single authoritative style source, so that style changes are predictable and do not get overridden by competing systems.

#### Acceptance Criteria

1. WHEN the application starts, THE Master_QSS SHALL be the sole stylesheet applied to QApplication
2. THE Legacy_Theme module `build_app_stylesheet()` function SHALL NOT be called anywhere in the codebase
3. WHEN a component previously relied on `build_ui_tokens()`, THE Design_Tokens `TokenRegistry.as_dict()` method SHALL provide equivalent values
4. IF the Legacy_Theme module is imported at runtime, THEN THE application SHALL raise a deprecation warning

### Requirement 2: Eliminate Inline setStyleSheet Calls from Views

**User Story:** As a developer, I want all widget appearance to be controlled by the global QSS via property selectors, so that theme changes propagate universally without per-widget overrides.

#### Acceptance Criteria

1. THE Master_QSS SHALL define Property_Selector rules for every visual role currently achieved through Inline_Style calls
2. WHEN a view file previously used `widget.setStyleSheet(...)`, THE view file SHALL instead use Style_Helper functions to assign the appropriate dynamic property
3. THE Master_QSS SHALL include property selectors for gradient-styled buttons (e.g., `QPushButton[uiRole="gradientPrimary"]`) to replace inline gradient QSS
4. THE Master_QSS SHALL include property selectors for the login screen tab bar (e.g., `QWidget[uiPanel="loginTabBar"]`, `QPushButton[uiRole="loginTabActive"]`, `QPushButton[uiRole="loginTabInactive"]`) to replace inline tab styling
5. THE Master_QSS SHALL include property selectors for branded panels (e.g., `QWidget[uiPanel="brandGradient"]`) to replace inline gradient backgrounds
6. WHEN the `apply_cta_button()` helper is called, THE Style_Helper SHALL assign a property role without calling `setStyleSheet()` on the button

### Requirement 3: Extend QSS Generator with Shell Roles

**User Story:** As a developer, I want the QSS generator to produce rules for all app shell components, so that the title bar, sidebar, and login screen are styled declaratively from tokens.

#### Acceptance Criteria

1. THE QSSGenerator SHALL produce Property_Selector rules for a custom title bar panel (`QWidget[uiPanel="titleBar"]`) with background color from Design_Tokens `surface_base`
2. THE QSSGenerator SHALL produce Property_Selector rules for title bar window control buttons (`QToolButton[uiRole="windowControl"]`) with transparent background, hover highlight, and a close-button danger variant (`QToolButton[uiRole="windowClose"]`)
3. THE QSSGenerator SHALL produce Property_Selector rules for the sidebar container (`QWidget[uiPanel="sidebar"]`) with fixed 200px width styling and background from Design_Tokens `surface_base`
4. THE QSSGenerator SHALL produce Property_Selector rules for sidebar nav items (`QPushButton[uiRole="navItem"]`) with icon+text layout, hover state, and an active state using a purple gradient pill background from Design_Tokens `secondary_accent`
5. THE QSSGenerator SHALL produce Property_Selector rules for the sidebar user profile section (`QWidget[uiPanel="userProfile"]`) positioned at the bottom of the sidebar
6. THE QSSGenerator SHALL produce Property_Selector rules for the login brand panel (`QWidget[uiPanel="brandGradient"]`) using a purple gradient from Design_Tokens `secondary_accent` to `secondary_accent_hover`
7. THE QSSGenerator SHALL produce Property_Selector rules for login form inputs at hero scale (`QLineEdit[uiField="loginInput"]`) with increased height (48px), larger font, and focus border using `secondary_accent_hover`

### Requirement 4: Rebuild Custom Title Bar

**User Story:** As a user, I want a polished custom title bar that matches the dashboard aesthetic, so that the frameless window feels intentional and premium.

#### Acceptance Criteria

1. THE Title_Bar SHALL display the application logo (SVG) and application name label on the left side
2. THE Title_Bar SHALL display a notification indicator button and window control buttons (minimize, maximize/restore, close) on the right side
3. WHEN the user drags the Title_Bar, THE App_Shell SHALL move the window position accordingly
4. WHEN the user double-clicks the Title_Bar, THE App_Shell SHALL toggle between maximized and normal window states
5. THE Title_Bar SHALL use a height of 40px with no visible border separating it from the sidebar and content area
6. THE Title_Bar SHALL be styled exclusively through Property_Selectors defined in the Master_QSS without any Inline_Style calls

### Requirement 5: Rebuild Sidebar Navigation

**User Story:** As a user, I want a sidebar that is visually consistent with the content area and provides clear navigation feedback, so that the interface feels unified and navigation state is always obvious.

#### Acceptance Criteria

1. THE Sidebar SHALL have a fixed width of 200px and use the same background color as the content area (Design_Tokens `surface_base`) to create a seamless boundary
2. THE Sidebar SHALL display navigation items as icon+text buttons with 12px body font and icon size of 20px
3. WHEN a navigation item is active, THE Sidebar SHALL display a purple gradient pill background behind the active item using Design_Tokens `secondary_accent` and `secondary_accent_hover`
4. WHEN the user hovers over an inactive navigation item, THE Sidebar SHALL display a subtle surface overlay background (Design_Tokens `surface_overlay`)
5. THE Sidebar SHALL display a user profile section at the bottom containing a user avatar placeholder and display name
6. THE Sidebar SHALL be styled exclusively through Property_Selectors defined in the Master_QSS without any Inline_Style calls

### Requirement 6: Rebuild Login Screen

**User Story:** As a user, I want a login screen that matches the premium NEXORA dashboard brand, so that the first impression of the application communicates quality and professionalism.

#### Acceptance Criteria

1. THE Login_Screen SHALL display a split-panel layout with a purple gradient brand panel on the left and a dark form panel on the right
2. THE Login_Screen brand panel SHALL use a gradient from `#4c1d95` through `#6d28d9` to `#a855f7` and display the application logo, headline text, and feature value propositions
3. THE Login_Screen form panel SHALL display a segmented tab control for switching between login and register forms with a purple gradient active pill indicator
4. THE Login_Screen form inputs SHALL use 48px height, 15px font size, and a focus border color of Design_Tokens `secondary_accent_hover`
5. THE Login_Screen submit buttons SHALL use a purple gradient background matching the brand panel gradient
6. THE Login_Screen SHALL include window control buttons (minimize, maximize, close) in the top-right corner of the form panel
7. THE Login_Screen SHALL be styled exclusively through Property_Selectors and the Master_QSS without any Inline_Style calls in the view file

### Requirement 7: Maintain Backward Compatibility

**User Story:** As a developer, I want existing views to continue rendering correctly after the style consolidation, so that the migration does not require rewriting all 20+ view files simultaneously.

#### Acceptance Criteria

1. THE Style_Helper functions (`set_panel_role`, `set_button_role`, `set_label_role`, `set_field_role`) SHALL retain their existing function signatures and behavior
2. THE Master_QSS SHALL include all existing `uiPanel`, `uiRole`, and `uiField` property selectors currently defined in the Legacy_Theme stylesheet
3. WHEN a view uses `set_button_role(button, "primary")`, THE Master_QSS SHALL style that button identically to the previous Legacy_Theme output
4. THE `TokenRegistry.as_dict()` method SHALL continue to return a flat dictionary compatible with the existing `build_ui_tokens()` return format
5. WHEN the application test suite runs, THE test results SHALL show all 156 existing tests passing without modification

### Requirement 8: Design Token Additions for Shell Components

**User Story:** As a developer, I want the design token system to include all color values needed for the new shell components, so that the purple brand gradient and sidebar active state are tokenized rather than hardcoded.

#### Acceptance Criteria

1. THE ColorTokens dataclass SHALL include a `nav_active_gradient_start` field with value `#7c3aed`
2. THE ColorTokens dataclass SHALL include a `nav_active_gradient_end` field with value `#a855f7`
3. THE ColorTokens dataclass SHALL include a `brand_gradient_start` field with value `#4c1d95`
4. THE ColorTokens dataclass SHALL include a `brand_gradient_mid` field with value `#6d28d9`
5. THE ColorTokens dataclass SHALL include a `brand_gradient_end` field with value `#a855f7`
6. THE ColorTokens dataclass SHALL include a `title_bar_bg` field with value matching `surface_base`
7. WHEN new token fields are added, THE TokenRegistry validator SHALL validate the new fields using the same hex-color format rules as existing fields
